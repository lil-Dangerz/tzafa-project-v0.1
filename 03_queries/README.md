# Query Arsenal

## Watch Role

Query Arsenal.

This folder stores important query logic, query references, validation status, and query-related workflow records used by the project.

It is not the verified truth ledger. It is not raw evidence storage.

## Purpose

Use `03_queries/` to preserve:

```text
query files
query references
query purpose
query status
validation notes
deprecation notes
supersession relationships
evidence and notebook links
```

Queries must not live only in chat.

If a query matters to project work, it should be stored or referenced here and registered in the query registry.

## Core Files

| File | Purpose |
|---|---|
| `QUERY_REGISTRY.md` | Register important queries, their purpose, status, and supporting references. |

## File Boundaries

### `03_queries/`

Use this folder for query logic and query records.

### `02_evidence/`

Use `02_evidence/` for proof artifacts, exports, screenshots, and logs.

Do not use query files as evidence storage.

### `PROJECT_STATE.md`

Use `PROJECT_STATE.md` for confirmed truth, not for storing query text.

### `04_notebooks/`

Use notebooks for exploration, execution, and analysis workflows.

If a notebook uses an important query, register the query here.

Use one of:

```text

tested

```

## Validation Rule

Do not mark a query as `validated` unless there is explicit confirmation from the user or supplied evidence showing that it works for its stated purpose.

Use:

```text
draft     = written but not executed
tested    = executed but correctness not confirmed
validated = explicitly confirmed correct for stated purpose
```

## Query Handling Rules

- Store important query logic in files or stable references.
- Register important queries in `QUERY_REGISTRY.md`.
- Link queries to supporting evidence, notebooks, reports, or decisions when possible.
- Record when a query is broken, deprecated, or superseded.
- Do not silently treat old or unvalidated queries as reliable.
- Prefer stable names that indicate purpose over vague temporary names.

## Query Review Flow

Queries should normally move through this pattern:

```text
write or capture query
store it or reference it in 03_queries/
register it in QUERY_REGISTRY.md
test or validate it
mark it as broken, deprecated, or superseded when needed
reference it from notebooks, reports, project state, or handoffs
```

The preferred prompt for query registration and classification is:

```text
prompts/query-review.prompt.md
```

That prompt may update only:

```text
03_queries/QUERY_REGISTRY.md
```

## Naming Guidance

When useful, organize queries by stable descriptive names.

Good examples:

```text
service-latency-check.esql
status-failures-by-project.sql
service-error-rate-trend.kql
critical-incident-filter.lucene
```

Avoid names like:

```text
query1.txt
new-query.esql
test.kql
working.sql
```

## Safety Rules

Do not store secrets, credentials, API keys, tokens, private keys, certificates, or restricted data in this folder unless explicitly sanitized and approved.

Do not treat a query as validated just because it was generated by an AI tool.

Do not let old chat memory override registered query status.
